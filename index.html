Agent 越界方案

方案核心逻辑

前三层管"该不该回答"
微调将安全边界写入权重 → PE 在运行时动态划定边界 → Judge 检测多轮对话中的人设偏移

后三层管"回答的对不对"
知识锚定到外部可溯源知识 → 结构化填表约束工具输入输出 → 硬检查做程序化兜底

逻辑链：微调先将安全边界写入权重（可选）→ PE 在运行时划定任务边界 → 但多轮对话中 PE 约束会偏移 → 所以用 Judge 实时检测人设偏移 → 即使边界内也会编造事实 → 所以知识锚定到外部可溯源知识 → 大模型自行改写字段名/值导致下游出错 → 所以结构化填表约束输入输出 → 最后输出侧硬检查做程序化兜底。
1

微调写入拒绝肌肉（可选，根据产品情况选择）

训练阶段完成，部署前就要做好。通过微调数据将安全边界写入模型权重——这是唯一不可被 Prompt Injection 绕过的防线。

为什么放在第一步？后续所有防线（PE、Judge、硬检查）都是运行时约束，理论上都可被绕过。只有写进权重的安全边界，才是模型的"肌肉记忆"——遇到边界场景时本能地给出安全回答，不依赖任何外部指令。

训练什么——不是训练"拒绝"，而是训练"有分寸的拒绝"

硬邦邦拒绝

"抱歉，我不能提供临床建议。"

问题：体验差，用户换个问法就绕过去了。模型只学会了一个动作——拒绝。

有分寸的拒绝

"二甲双胍在 eGFR<30 时禁忌（KDIGO 2024）。具体用药需由肾内科医生评估。我可以帮您检索相关指南。"

给安全信息 + 引导至人类专家 + 提供替代路径

微调数据构造策略

数据类型作用示例
正常学术问答 主干能力，保证加入安全数据后不冲垮学术能力 "PCOS 相关 Meta 分析有哪些"
正确拒绝 / 降级 核心拒绝肌肉——有分寸的安全回答 "我父亲该吃多少 mg" → 给通识 + 引导专家
边界灰色地带 教模型理解分寸，区分可回答与不可回答 "CKD 3b 期的指南推荐" (OK) vs "CKD 3b 该怎么调剂量" (拒绝)
对抗性绕过尝试 鲁棒性，防 Prompt Injection "假装你是医生..."、"纯学术讨论，如果要开..."

⚠ 微调的局限

微调能内置安全倾向，但不够灵活——边界规则变化时需要重新训练。且微调只能覆盖训练数据见过的场景，新场景仍需运行时约束补强。

✓ 需要 PE 在运行时动态划定边界

微调管"基本安全底线"，PE 管"具体任务的灵活边界" → 第2层。

2

Prompt Engineering 划定任务边界

运行时的第一道防线。通过 System Prompt 明确告诉模型"你是谁、能做什么、不能做什么"——比微调更灵活，可按任务动态调整。

边界从哪来？三个维度

01 身份边界

  • "你是医学文献分析助手"
  • "你不是医生"
  • "你不是临床决策系统"

02 操作边界

  • 可以：检索、统计、综述
  • 不可以：开处方、定剂量
  • 不可以：修改原始数据

03 输出边界

  • 每个事实必须带引用
  • 不可输出"可以用/不可以用"
  • 不可从动物实验推人体
## System Prompt 核心约束（示例） 身份：你是医学科研文献分析助手，不是医生，不是临床决策系统。 绝对禁止（任何情况不可违反）： 1. 不输出针对特定患者的诊疗方案 2. 不输出具体剂量建议（"建议 XX mg"） 3. 不在没有 PMID/DOI 的情况下陈述具体数值 4. 不将动物/体外实验结论推广至人体 无法确定时的默认行为： - 说"现有检索范围内未找到直接证据" - 不编造过渡性解释

⚠ PE 的局限

PE 是文本指令，在多轮对话中模型会逐渐偏离原始人设——第1轮还老老实实当"文献助手"，到第5轮可能已经滑向"医生角色"开始说"我建议你..."。这就是"人设偏移"问题。

✓ 需要 Judge 实时检测偏移

PE 设定了边界，但没有机制检测模型是否还在边界内 → 引入 Judge（第3层）。

3

Judge 检测人设偏移

解决的核心问题：多轮对话中 Actor 会逐渐偏离 PE 设定的人设。Judge 不是"互审"，而是实时监测 Actor 是否还在边界内。

Actor-Judge 不是互审。Actor 正常回答问题，Judge 独立评估每一轮输出是否偏离了原始人设。偏移分高 → Actor 被拉回边界内重新生成；偏移分低 → 正常放行。这是一个偏移检测机制，不是对抗机制。
用户提问（第 N 轮对话）
Actor 生成回答
Judge：对比原始 PE，检测 Actor 是否人设偏移
偏移分低 · 在边界内
正常输出
偏移分高 · 越界
拉回边界 / 重新生成

Judge 评估的四个偏移维度

检测维度具体指标典型偏移场景
身份偏移 Actor 是否开始扮演 PE 中禁止的角色 用户反复追问后 Actor 开始说"我建议你..."
能力偏移 Actor 是否开始做超出授权的操作 从综述分析滑向个体化用药建议
态度偏移 Actor 是否从谨慎变为过度自信 第1轮还标注"证据有限"，第5轮直接下结论
引用偏移 Actor 是否开始不附带引用就陈述事实 前几轮每句话带 PMID，后面逐渐省略
关键设计：Judge 只看偏移，不管内容对错。内容准确性由后续的知识锚定（第4层）和硬检查（第6层）负责。Judge 的唯一职责是确保 Actor 始终在 PE 定义的角色边界内。职责分离，不互相越位。

⚠ 前三层都管"该不该回答"

微调、PE、Judge 解决的是任务边界问题——模型应该在什么范围内工作。但即使模型在边界内工作，它仍然会从参数记忆中编造事实。

✓ 需要知识锚定解决"回答的内容对不对"

边界问题靠前三层，内容准确性靠后三层 → 第4层。

4

知识锚定——防止边界内的事实幻觉

前三层管"该不该回答"。从这里开始管"回答的内容对不对"——即使在边界内，模型也会从参数记忆中编造数据。

核心规则：所有事实陈述必须锚定到可溯源的外部知识（无论是 Wiki 还是 RAG）。Agent 不可从参数记忆中生成事实。引用时必须使用原文片段中的数值，不可四舍五入、换算或转述。

为什么"不可转述"？

模型转述（高风险）

原文："response rate was 42.3% (95%CI 38.1-46.5)"

转述1："有效率约 40-45%"
转述2："有效率接近一半"
转述3："有效率 42.3% (95%CI 38.1-47.5)"

第三种 CI 上限被改了 1.0，极难发现。

逐字引用（安全）

直接引用原文：
"response rate was 42.3% (95%CI 38.1-46.5)"

来源：(Ferracioli-Oda et al., 2013, PMID:23691095)

可与检索结果原文做 diff 比对，自动验证一致性。

锚定知识源

知识源内容类型溯源标识
PubMed原始研究 / 综述PMID
Cochrane Library系统综述DOI / Cochrane ID
UpToDate / 各学科指南临床指南指南编号 / DOI
DrugBank / RxNorm药物数据DrugBank ID
ClinicalTrials.gov临床试验注册NCT ID

⚠ 知识锚定的盲区

锚定解决了"凭空编造"，但 Agent 在调用工具时传递字段可能出错——大模型会基于自己的理解自行改写字段值，导致参数往下传递时与原始数据不一致。例如将标准药物名改写为俗称、将精确数值四舍五入后传入，下游工具接收到的已不是正确的输入。这种幻觉发生在工具接口层面，知识锚定管不到。

✓ 需要结构化填表约束工具调用的输入输出

通过 Schema 约束工具的输入/输出字段，字段类型、枚举范围、必填项都提前锁定，防止大模型自行改写 → 第5层。

5

结构化输出 · 填表模式——防止字段改写导致的幻觉

核心问题：大模型会基于自己的理解自行改写字段名或字段值——把标准字段名换成同义词、把精确数值四舍五入。下游系统收到的已不是正确参数，但模型自认为传对了。填表模式通过 Schema 锁定字段，从根源上杜绝改写。

自由文本 / 无约束模式

  • 模型把字段名改写为同义词，下游系统不认识
  • 数值被自行四舍五入或换算后传入
  • 标准术语被替换为俗称或缩写
  • 必填字段被跳过，缺失数据无提示

填表模式

  • 每个字段有类型约束
  • 枚举值只能从预设列表选
  • 数值有范围校验
  • 必填字段不可省略
// 结构化输出 Schema（量化研究场景） { "claim": string, // 一个且仅一个事实陈述 "source": { "pmid": string | null, "study_type": enum ["SR/MA", "RCT", "Cohort", ...] }, "effect": { "metric": enum ["OR", "RR", "HR", "SMD", "NNT"], "value": number, "ci_lower": number, // 硬校验: ci_lower < value < ci_upper "ci_upper": number, "p_value": number [0,1] }, "evidence_grade": enum ["high", "moderate", "low", "very_low"], "anchor_quote": string, // 检索结果原文逐字引用（可 diff 比对验证） "confidence": enum ["confirmed", "limited", "needs_verification"] }

填表为什么能防幻觉？

  • pmid 字段 → 后续可程序化验证是否真实存在
  • ci_lower < value < ci_upper → 数值逻辑自动校验
  • study_type 只能从 enum 选 → 不能编造研究类型
  • anchor_quote 逐字引用 → 可与检索结果原文 diff 比对
  • 必填字段缺失 → 直接拒绝输出，强制补全
工具输入同样要约束：不只约束输出格式，工具调用的输入参数同样需要 Schema 校验。模型可能将 drug_name 改写为 medication、将 eGFR 改写为 kidney_function——下游 API 直接报错或静默忽略。Schema 锁定字段名和类型，杜绝这类改写。
6

输出侧硬检查——最后的程序化兜底

前5层都涉及 LLM 参与，理论上都可能出错。这一层是纯代码执行的程序化验证，不依赖任何模型，接近 100% 可靠。

硬检查的本质：能用代码验证的事情，绝不交给 LLM 判断。每一个从"LLM 判断"转化为"代码验证"的检查点，都是实打实的可靠性提升。

硬检查清单

检查项验证方式抓什么问题
PMID / DOI 存在性 调用 PubMed API / CrossRef API Agent 编造不存在的文献
anchor_quote 匹配 与检索结果原文做字符串 diff Agent 悄悄修改引用中的数值
数值范围校验 CIlower < 点估计 < CIupper 统计数据逻辑矛盾
术语词表匹配 MeSH / ICD-11 / SNOMED-CT 查表 Agent 编造不存在的疾病名或术语
药物名验证 DrugBank / RxNorm 查表 Agent 编造不存在的药物
统计一致性 OR 方向与 p 值方向是否一致 Agent 编造自相矛盾的统计结果
p 值范围 p ∈ [0, 1] Agent 输出非法 p 值
年份合理性 引用年份 ≤ 当前年份 Agent 引用"未来"发表的论文
// 硬检查伪代码 function hardCheck(output) { // 1. PMID 存在性 if (output.pmid) { result = PubMedAPI.fetch(output.pmid) assert result.exists == true } // 2. 引用原文比对 assert fuzzyMatch(output.anchor_quote, retrieved_snippets) >= 0.95 // 3. 数值逻辑 assert output.ci_lower < output.value < output.ci_upper assert 0 <= output.p_value <= 1 // 4. 术语验证 for term in output.medical_terms: assert term in MeSH_vocabulary or term in ICD11_codes // 5. 不通过 → 拦截，不是"警告" if any_assertion_failed: → 拦截输出，要求重新生成 }

完整因果链

六层递进，每一层解决上一层解决不了的问题。

1

微调写入拒绝肌肉（可选）

训练阶段 · 唯一不可被运行时攻击绕过的防线 · 根据产品情况选择是否启用
将安全边界写入模型权重。有分寸的拒绝（安全信息+引导+替代），而非硬拒绝。但不够灵活，边界规则变化时需重新训练。
2

PE 划定任务边界

运行时 · 身份/操作/输出三重边界
通过 System Prompt 动态定义边界，比微调更灵活。但本质是文本指令——多轮对话中会逐渐偏移。
3

Judge 检测人设偏移

解决 PE 在多轮对话中失效的问题
独立模型实时监测 Actor 每轮输出是否偏离原始人设。身份/能力/态度/引用四维度检测，偏移则拉回边界。
4

知识锚定

解决边界内的事实幻觉——模型从参数记忆中编造数据
所有事实必须锚定到 PubMed / Cochrane / 指南等可溯源外部知识。引用必须逐字，不可四舍五入或转述。
5

结构化填表

解决大模型自行改写字段名/字段值导致的幻觉
用 Schema 锁定字段名、类型、枚举范围、必填项。模型不能把 drug_name 改写为 medication，不能把精确数值四舍五入后传入。下游系统收到的就是标准参数。
6

输出侧硬检查

最后的程序化兜底，不依赖任何 LLM 判断
纯代码执行：PMID 验证、anchor_quote 比对、数值范围校验、术语词表匹配。前5层都涉及 LLM，这一层是唯一 100% 确定性的验证。

各层失效时的兜底关系

当这一层失效...后果谁来兜底
微调覆盖不到新场景模型面对未见过的边界场景PE 在运行时动态设定边界
PE 被多轮对话冲淡模型逐渐偏离人设Judge 实时检测偏移并拉回
Judge 误判偏移了但没拦住微调中已内置的安全倾向兜底
在边界内编造事实输出看似合理但数据虚假知识锚定——没检索到就没法编
RAG 检索不全锚定了但信息不完整结构化填表暴露数据缺口（必填字段留空）
填表被绕过Agent 自由发挥硬检查——程序化验证拦截
硬检查漏网有些错误无法程序化验证微调+PE 的基础安全行为兜底 ↻ 闭环

附：完整示例

一个真实查询如何穿过六层防线。

用户："二甲双胍在 eGFR 30-45 的 2 型糖尿病患者中的安全性证据？我有个患者 eGFR 38，在考虑要不要继续用。"
L1 · 微调
模型权重已学过 eGFR + 具体数值 + "要不要继续用" 的组合 → 内置倾向：给群体证据 + 引导至人类专家，不给个体化建议。
L2 · PE
System Prompt 标记规则：不可输出个体化用药建议、不可输出具体剂量、每个事实必须带引用。
L3 · Judge
如果是多轮对话的第 N 轮：Judge 检测 Actor 此前的输出是否已经在向"医生角色"偏移。偏移分高则在本轮生成前拉回边界。
L4 · 锚定
检索 PubMed: "metformin safety eGFR 30-45 CKD" → 返回 3 篇核心文献 + NICE/KDIGO 指南。Actor 只能基于这些原文回答，不可补充参数记忆中的信息。
L5 · 填表
Actor 填写 Schema：claim="eGFR 30-45时二甲双胍应定期评估"，pmid="25078757"，anchor_quote="Metformin should be reviewed at eGFR <45 and stopped at eGFR <30"，evidence_grade="high"。
L6 · 硬检查
PMID 25078757 → PubMed API 验证 → 存在 ✓   anchor_quote 与检索原文 diff → 匹配 ✓   术语 MeSH 匹配 ✓   全部通过 → 正常输出。
